# Ballers & Bookworms

A private web application for Ballers & Bookworms student athletes. The app provides three core features: a fund request and budgeting tool, an AI-powered mentor matching system, and an NCAA eligibility checker that processes transcripts via Claude vision.

## 1. Project Overview

Ballers & Bookworms is a private portal for student athletes with three features:

**Fund Requests & Budgeting**
Student athletes can submit funding requests across eight categories (academic supplies, athletic equipment, tutoring, athletic training, nutrition consulting, camp fees, travel costs, other) up to $1,000 per year. Staff review requests and can approve, deny, flag, or mark them as reimbursed. Athletes see their remaining balance and full request history in real time.

**Mentor Matching**
Athletes describe what they are looking for in a mentor in free text. An AI pipeline extracts their needs, filters the mentor pool by sport and availability, embeds the request using OpenAI's text-embedding-3-small model, runs cosine similarity search against pre-computed mentor embeddings stored in pgvector, and returns ranked matches with personalized explanations generated by Claude. Athletes can then request a session directly from the app.

**NCAA Eligibility Checker**
Athletes upload a high school transcript (JPEG or PDF). Claude vision extracts the school name and CEEB code, the app scrapes the NCAA High School Portal for that school's approved course list, then Claude maps every course against the approved list, converts numeric grades to letter grades using the school's own printed scale or standard 100-point cutoffs, and calculates core-course GPA and DI/DII eligibility status including the 10/7 rule.

## 3. Architecture Overview

```
Browser (React + Vite)
    │
    ├── Supabase Auth (GitHub OAuth)
    ├── Supabase PostgREST  ──→  Postgres (RLS enforced)
    ├── Supabase Storage    ──→  transcripts bucket (private)
    │                       ──→  demo-transcripts bucket (public)
    └── Supabase Edge Functions (Deno)
            ├── find-mentors          ──→  OpenAI Embeddings API
            │                         ──→  Claude Haiku (explain matches)
            │                         ──→  match_mentors() SQL function (pgvector)
            ├── embed-mentor          ──→  Supabase built-in AI (gte-small)
            ├── request-session       ──→  Resend (email)
            ├── scrape-ncaa-courses   ──→  NCAA HS Portal (web scraping)
            └── process-transcript   ──→  Claude Haiku (vision/document)
                                      ──→  scrape-ncaa-courses (internal call)
```

**How the pieces fit together:**

All database access from the browser goes through Supabase PostgREST with Row Level Security enforced at the Postgres layer. No sensitive operations — AI calls, web scraping, email — happen in the browser. These all live in Deno-based Edge Functions that run server-side and are called via authenticated HTTP POST from the frontend.

The `process-transcript` function is the most complex: it makes two Claude calls (a cheap Pass 1 to extract the school name, then a full Pass 2 to map all courses), calls `scrape-ncaa-courses` as an internal HTTP request to fetch the approved course list, and saves the results to `eligibility_assessments` and `eligibility_courses`.

Mentor embeddings are pre-computed (not generated at query time). When a mentor is inserted or updated, the `embed-mentor` function generates a 1536-dimension vector and stores it in `mentor_embeddings`. At query time, `find-mentors` embeds the athlete's request with the same model and calls the `match_mentors()` Postgres function to run cosine similarity ranking entirely inside the database.

## 6. Edge Functions

All functions live in `supabase/functions/`. Each is written in TypeScript for the Deno runtime. Deploy any function with:

```bash
npx supabase functions deploy <function-name>
```

To target a specific project without re-linking:

```bash
npx supabase functions deploy <function-name> --project-ref <project-ref>
```

### `embed-mentor`
**Trigger:** POST from seed scripts or database webhook (mentor INSERT/UPDATE)
**Purpose:** Generates a 1536-dimension embedding for an active mentor profile using Supabase's built-in AI (gte-small model) and stores it in `mentor_embeddings`. Called once per mentor when they are created or their profile changes.
**Extra secrets:** None (uses built-in Supabase AI)

### `find-mentors`
**Trigger:** POST from browser
**Purpose:** Full AI mentor matching pipeline. Receives the athlete's free-text request, calls Claude Haiku to extract structured needs (sport, goals, challenges), filters the mentor pool, embeds the request using OpenAI's text-embedding-3-small, runs cosine similarity search via the `match_mentors()` Postgres function, then calls Claude again to generate a personalized explanation for each match.
**Extra secrets:** `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`

### `request-session`
**Trigger:** POST from browser
**Purpose:** Inserts a `session_requests` row and sends a notification email to the mentor via Resend.
**Extra secrets:** `RESEND_API_KEY` (optional), `FROM_EMAIL` (optional)

### `scrape-ncaa-courses`
**Trigger:** POST from browser or internal call from `process-transcript`
**Purpose:** Scrapes the NCAA High School Portal (`web3.ncaa.org/hsportal`) for a school's NCAA-approved course list. Establishes a portal session via GET, then POSTs a search by CEEB code (preferred) or school name. Caches results in `ncaa_approved_courses_cache` for 30 days. Returns `found`, `not_found`, or `multiple_matches`.
**Extra secrets:** None

### `process-transcript`
**Trigger:** POST from browser
**Purpose:** Two-pass transcript analysis pipeline. Pass 1 (cheap): Claude Haiku extracts the high school name, state, and CEEB code from the transcript image or PDF. Pass 2 (full): Claude Haiku maps every course against the school's NCAA-approved list, converts grades to letter grades using the school's own printed scale or standard 100-point cutoffs (90=A, 80=B, 70=C, 65=D, <65=F), calculates core-course GPA (excluding in-progress courses), and checks DI/DII eligibility including the 10/7 rule. Results are saved to `eligibility_assessments` and `eligibility_courses`. Pass 2 runs at temperature=0 for deterministic output.
**Extra secrets:** `ANTHROPIC_API_KEY`

## 7. Database Schema

The schema is defined entirely in `supabase/migrations/`. Every table has Row Level Security enabled. The `is_staff()` helper function (defined in migration 002) is used across all RLS policies to grant staff full read/write access.

### Core tables

**`student_athletes`**
One row per athlete, linked to `auth.users`. Stores name, school, grade, sports, and hometown. The central entity that most other tables reference.

**`staff_users`**
Email allowlist for staff accounts. The `is_staff()` function checks this table to determine whether the calling user has staff privileges.

**`funding_requests`**
Fund requests submitted by athletes. References `student_athletes` and `auth.users`. Tracks category, amount, description, status (`pending`, `approved`, `reimbursed`, `denied`, `flagged`), and an optional staff note.

### Mentorship tables

**`mentors`**
One row per mentor. Includes sport, college, division, current job, bio, and status (`pending`, `active`, `inactive`). Staff must set status to `active` before a mentor appears in search results.

**`mentor_mentorship_areas`**
Many-to-one with `mentors`. Each row is one area/topic the mentor can help with (e.g. "SAT prep", category "academic").

**`mentor_availability`**
One-to-one with `mentors`. Hours per week, format (in-person/virtual/either), and timezone.

**`mentor_embeddings`**
One-to-one with `mentors`. Stores the 1536-dimension vector used for similarity search. Only accessible to service role and staff.

**`mentee_intake`**
Records the athlete's free-text request and structured preferences submitted when searching for a mentor.

**`matches`**
Produced by the matching pipeline. Stores mentor/mentee pairs, cosine similarity score, and match explanation. Status progresses from `pending_review` to `confirmed`, `declined`, or `completed`.

**`session_requests`**
Created when an athlete clicks "Request a Session." Triggers a notification email to the mentor.

### NCAA eligibility tables

**`eligibility_assessments`**
One row per transcript upload. Stores the school name, CEEB/ncaa_school_code, overall eligibility status, core course GPA, total core credits, and pre-7th-semester credits.

**`eligibility_courses`**
One row per course extracted from the transcript. References `eligibility_assessments`. Stores course name, mapped NCAA category, grade, credit (0 for in-progress), quality points, approval status, confidence level, and a `needs_review` flag.

**`ncaa_approved_courses_cache`**
Caches the NCAA portal's approved course list per school for 30 days. Written only by `scrape-ncaa-courses` via service role. Keyed on `ncaa_school_code`.

### Key Postgres function

**`match_mentors(query_embedding, candidate_ids, match_count)`**
A `SECURITY DEFINER` function that runs cosine similarity search using the `<=>` pgvector operator. Only callable by service role. Called by `find-mentors` after filtering and embedding the athlete's request.

## 8. Key Design Decisions

**Claude vision for transcript parsing instead of a structured form**
Athletes upload a transcript image or PDF rather than entering course data manually. This dramatically improves UX but introduces AI variability. Several debugging sessions were required to make grade conversion deterministic: the Pass 2 Claude call runs at temperature=0, the system prompt includes an explicit 4-step grading scale decision tree with hard numeric cutoffs for 100-point scales, and the prompt instructs Claude to use the transcript's own printed scale legend when one is present.

**Two-pass Claude architecture for transcript processing**
Transcript processing is split into a cheap Pass 1 (school name extraction only, 256 max tokens) and a full Pass 2 (complete course mapping, 8192 max tokens). This makes the school confirmation step possible — without Pass 1 returning early, there would be no way to show the user the extracted school name and let them correct it before spending tokens on the full analysis.

**Live NCAA portal scraping instead of a static dataset**
The app scrapes `web3.ncaa.org/hsportal` at request time rather than maintaining a pre-built copy of approved courses. Results are cached in Postgres for 30 days. The benefit is always-current data; the cost is fragility — the portal uses Akamai bot protection, requires session cookies established via a specific GET URL, and the POST search body must include `hsActionSubmit=Search` as a form field (not a query parameter). These details required significant reverse-engineering.

**pgvector for mentor matching instead of keyword filtering**
Mentor matching uses embedding similarity rather than exact field matching. This allows semantic matches — an athlete asking for help with "balancing school and training" can match a mentor who lists "time management" as an area without any keyword overlap. The tradeoff is infrastructure complexity: pgvector, the `embed-mentor` deployment, and the pre-computation pipeline.

**RLS as the sole security boundary**
All data isolation is enforced through Postgres Row Level Security rather than application-level checks. Security lives at the database layer and cannot be bypassed by a misconfigured edge function or a direct API call. The `is_staff()` helper function centralizes the staff privilege check so it is consistent across all policies.

**Supabase Edge Functions instead of a separate backend**
All server-side logic lives in Deno-based Edge Functions on Supabase rather than a separate Node/Express server. This keeps everything on one platform (database, auth, storage, compute) and simplifies deployment. The tradeoff is the Deno runtime (different module imports, no npm), cold starts, and a 60-second execution limit that required careful timeout management in `process-transcript`.

## 10. Known Limitations

**NCAA portal scraper fragility**
The scraper depends on specific session cookie behavior, a POST body format that was reverse-engineered from the portal's form submission, and HTML structure that could change at any time. The portal uses Akamai bot protection that may begin blocking server-side requests at scale. A more robust long-term solution would be a scheduled job that pre-fetches approved course lists nightly rather than scraping on demand.

**Transcript processing is synchronous**
`process-transcript` runs as a single synchronous HTTP request that must complete within Supabase's 60-second edge function limit. At scale this is unreliable — Claude vision calls, the NCAA scrape, and two round trips add up. The correct architecture is an async job queue where the upload returns immediately and the UI polls for results.

**No automated tests**
There is no test suite anywhere in the codebase. The GPA calculation, eligibility rule logic, and course mapping in `process-transcript` are particularly high-risk — a silent regression could give athletes incorrect eligibility information. Unit tests with known transcript inputs and expected outputs are needed before this feature is used in a production context.

**Mentor embeddings have no recovery mechanism**
If `embed-mentor` fails silently during seeding (e.g. a transient error), the mentor has no embedding and will never appear in search results. There is no monitoring for mentors missing embeddings and no scheduled backfill job. Running `npm run embed:mentors` manually will re-generate any missing embeddings.

**No rate limiting on edge functions**
Any authenticated user can call `process-transcript` or `find-mentors` unlimited times, which could run up significant Anthropic and OpenAI API costs. Per-user rate limiting and spending caps should be added before opening the app to a large user base.

**Fund request workflow is minimal**
The current approval flow is a single status field with no multi-step approvals, no document/receipt linking to specific requests, no budget rollover between academic years, and no audit log of who approved what and when.

**No admin tooling**
There is no super-admin interface for managing the platform — adding or removing staff, adjusting per-athlete budget limits, viewing platform-wide analytics, or moderating mentor applications all require direct database access.
